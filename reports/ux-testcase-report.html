Criterio A2UI oficial aplicado

Obligatorio

Se reviso la documentacion oficial de A2UI y se aplico este limite al caso: la UI viaja como datos declarativos, no como codigo; el agente genera componentes del catalogo; el renderer usa componentes nativos; y el backend se mantiene como frontera de validacion, normalizacion de variantes seguras, reparacion/fallback y trazabilidad de datos. No se anadio un parser de intencion conversacional ni una conversion backend de texto a UI enriquecida.

  • A2UI Protocol v0.9.1: version de produccion actual, mensajes createSurface, updateComponents, updateDataModel y deleteSurface.
  • A2UI overview: formato declarativo, componentes preaprobados del catalogo y sin ejecucion arbitraria.
  • A2UI actions: acciones como event o functionCall, sin handlers ad hoc.
  • A2UI data flow: agente -> transporte -> renderer -> UI nativa.

Problemas encontrados

  • En mobile, el primer viewport no dejaba el CTA visible para una situacion de 8% de bateria.

Cambios realizados: Caso 002

Acotados al caso
  • Se ajustaron instrucciones del agente para bateria <=10%: una parada primaria, riesgo humano visible cerca de la parada y solo 1-2 alternativas trazadas despues.
